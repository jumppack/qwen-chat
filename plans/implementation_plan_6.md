# Model Switching Implementation Plan (Plan 6)

This plan covers the transition from a single hardcoded model to a dynamic, memory-conscious multi-model architecture.

## User Review Required

> [!IMPORTANT]
> - **Database Update**: This plan involves adding a `model` field to the `Chat` table in `schema.prisma` to persist the selected model per chat. A migration (`npx prisma db push`) will be required.
> - **Model Names**: The models to be configured are `qwen2.5-coder:32b` (Qwen 2.5) and `qwen2.5:27b` (assuming Qwen 3.6 27B translates to this or similar in Ollama; please confirm the exact Ollama tag for "Qwen 3.6 27B").

## Proposed Changes

### 1. Database Layer

#### [MODIFY] [schema.prisma](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/prisma/schema.prisma)
- Add a `model` field to the `Chat` model with a default value (e.g., `qwen2.5-coder:32b`). This ensures that returning to an old chat remembers which model was used.

### 2. Backend Architecture: The "Graceful Swap"

#### [NEW] [route.js (model-switch)](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/src/app/api/model/switch/route.js)
- Create a new endpoint `POST /api/model/switch`.
- **Logic**:
  1. Receive `currentModel` and `newModel` in the request body.
  2. Send a dummy request to Ollama for the `currentModel` with `keep_alive: 0` to unload it from RAM immediately.
  3. Send a dummy request to Ollama for the `newModel` with `keep_alive: "5m"` (or similar) to load it into RAM.
  4. Respond with a success status once the new model is loaded.

#### [MODIFY] [route.js (chat)](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/src/app/api/chat/route.js)
- Update the streaming logic to use the model selected by the user (passed in the request payload) rather than the hardcoded `qwen2.5-coder:32b`.

#### [MODIFY] [route.js (chats API)](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/src/app/api/chats/route.js)
- Ensure the `POST` request to create a new chat accepts and saves the initial `model` selection.

### 3. Frontend UI: The ChatGPT-style Selector

#### [MODIFY] [page.js](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/src/app/page.js)
- **State Management**: Add `activeModel` and `isModelSwitching` states.
- **UI**: Add a floating dropdown centered at the top of the chat area to select between models.
- **Halted State**: When `isModelSwitching` is true:
  - Show a loading spinner/indicator in the dropdown.
  - Disable the message `textarea` and send button.
- **Logic**: 
  - `handleModelChange`: Triggers when a new model is selected. Updates the chat record in the database, calls `/api/model/switch` to manage memory, and sets `isModelSwitching` appropriately.

#### [MODIFY] [globals.css](file:///Users/karan/Desktop/Code/qwen3_5_model_prompting/src/app/globals.css)
- Add styles for the model selector dropdown, ensuring it matches the premium glassmorphic theme.

## Verification Plan

### Automated/Manual Testing
1. **RAM Monitoring**: Open Activity Monitor and verify that RAM usage drops when the switch starts (Unload) and rises only when the new model is ready (Load).
2. **UI Locking**: Ensure the input area becomes "read-only" and disabled during the swap.
3. **Persistence**: Refresh the page after a switch and verify the dropdown still reflects the newly selected model for that specific chat.
4. **Chat Execution**: Send a message after swapping and confirm the response is generated by the newly selected model.
